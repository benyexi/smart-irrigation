## 0. 项目背景

### 0.1 发起人与学术背景

本项目由**北京林业大学人工林水分关系实验室（PWRlab）**发起，负责人为席本野教授（博士生导师，国家万人计划青年拔尖人才）。席老师长期从事人工林水分关系、根系结构与功能、人工林高效培育理论与技术研究，在全国建有 **22 个 CP-GPE Net 野外监测站点**，积累了超过 11 年的树木水分生理实测数据（液流、茎径、叶片膨压、土壤水势等）。

### 0.2 项目起源

2026 年 3 月底，席老师委托 AI 开发助手（Dispatch/Claude/Codex）从零搭建本平台。开发过程经历了以下几个阶段：

1. **需求讨论阶段**：通过 Dispatch（Claude）完成需求梳理，确定核心功能模块和技术栈；
2. **Manus 搭架子**：用积分驱动 Manus 完成前端页面初始框架（React + Ant Design + ECharts，纯静态演示版）；
3. **Claude/Codex 接力开发**：完成深色主题重设计、站点管理、田块可视化编辑器、MQTT 实时通信、IoT 接入文档等核心模块；
4. **当前状态**：已具备完整"看板-监控-站点配置-IoT接入-产品介绍"流程的前端原型平台。

### 0.3 核心产品定位

**智灌云**是一个面向果树、林木、农作物的智能灌溉决策 SaaS 平台，计划与硬件合作方**时域通**联合推向市场。

**核心差异化**：竞品（Netafim、华为、阿里云平台等）普遍以土壤墒情驱动灌溉决策，智灌云的核心卖点是**植物生理指标（树液流速率、茎干直径变化量、叶片膨压）+ 土壤水势联合驱动**，真正从植物需水端出发。

这一差异背后是席老师团队 11 年的野外实测数据积累和 50+ 篇 SCI 论文支撑，是竞品无法快速复制的技术壁垒。

### 0.4 商业模式设想

- 软件订阅（SaaS）+ 硬件套卖（与时域通合作）
- 目标客户：林场/国有林业局、高端果树种植户（苹果/梨/葡萄）、科研院所、三北工程实施方
- 主攻场景：高价值作物（果农对水分管理敏感、溢价空间大）、人工林（席老师自有数据最强）

## 1. 项目定位与当前形态

本项目是一个**前端主导的智能灌溉平台原型/演示系统**，目标是逐步演进为可商业化的灌溉 SaaS 产品。当前形态如下：

- 技术实现：React 19 + TypeScript + Vite + Ant Design 6 + ECharts。
- 部署方式：GitHub Pages 静态部署（HashRouter + `base=/smart-irrigation/`）。
- 数据现状：
  - 业务核心页面大部分仍使用 `src/mock/` 作为展示数据源。
  - 站点管理（`/sites`）与站点切换当前走 `siteRepository + localStorage`，已具备后端替换接口位。
  - 实时链路采用 MQTT（浏览器内可收发闭环演示）。
- 产品阶段：已具备完整"看板-监控-站点配置-IoT接入"流程，但**仍未接入真实后端/数据库/设备鉴权体系**。

## 5. 核心代码机制

### 5.1 路由与鉴权

- `src/App.tsx` 使用 `HashRouter`，适配 GitHub Pages 静态部署。
- `RequireAuth` 控制除 `/login` 外所有页面的访问。
- `AuthProvider` 将用户信息保存在 `localStorage` + `sessionStorage`。
- 当前路由已改为 `React.lazy + Suspense` 做页面级代码分包，首屏不再一次性加载 Dashboard、Monitor、Knowledge、Screen 等重页面逻辑。

### 5.2 站点数据模型

- `src/types/site.ts` 是当前业务核心模型。
- `Site` 下挂：`sensors`、`pipelines`、`modeParams`、`alarmRules`。
- 后续后端建模应优先对齐该结构，避免前后端字段割裂。

### 5.3 站点存储层

当前站点数据已分为三层：

- `src/repositories/siteRepository.ts`
  - 对上提供 `getSnapshot / listSites / getCurrentSiteId / setCurrentSiteId / saveSite / deleteSite / subscribe`
  - 当前默认实现为 `localSiteRepository`
  - 后续替换真实后端时，优先替换这一层
- `src/stores/siteStore.ts`
  - 统一维护运行时 `sites + currentSiteId`
  - 对页面暴露保存、删除、切站点动作
  - 通过 repository 订阅存储变化并回刷全局状态
- `src/utils/siteStorage.ts`
  - 仍负责当前的 `localStorage` 持久化和跨页面同步事件
  - `getSites()` 首次调用自动初始化 3 条 demo 站点并写入 localStorage
  - `saveSite(site)` / `deleteSite(id)` / `setCurrentSiteId(id)` 负责真实落盘

这意味着当前“伪后端”主线已不再是页面直接碰 `localStorage`，而是统一经过 repository 与 store。

### 5.4 MQTT 客户端封装

`src/utils/mqttClient.ts` 提供统一能力：

- `connectMqtt(url)` / `disconnectMqtt()`
- `subscribeMqtt(topic, handler)` / `unsubscribeMqtt()`
- `publishMqtt(topic, payload)`
- `topicMatches(filter, topic)` 支持 `+` 与 `#` 通配符匹配
- `getMqttStatus()` / `addMqttStatusListener()`

**Topic 规范**（已在代码中统一使用）：

| Topic | 方向 | 说明 | QoS |
|---|---|---|---|
| `siz/v1/{siteId}/sensor/{deviceId}/data` | 设备→平台 | 传感器数据上报 | 0 |
| `siz/v1/{siteId}/control/{deviceId}/cmd` | 平台→设备 | 控制指令下发 | 1 |
| `siz/v1/{siteId}/control/{deviceId}/ack` | 设备→平台 | 指令执行回执 | 1 |
| `siz/v1/{siteId}/status` | 设备→平台 | 设备心跳状态 | 0 |

当前使用公共 Broker：`wss://broker.emqx.io:8084/mqtt`，生产环境须替换为私有部署并增加鉴权。

配套 Hook：

- `src/hooks/useMqttSubscription.ts`：将订阅/退订绑定到 React 生命周期，减少手动忘记清理的风险。
- `src/hooks/useMqttStatus.ts`：统一状态监听逻辑，避免页面各自直接持有监听器。

### 5.5 图表与构建性能优化

- `src/components/ECharts/ReactECharts.tsx` 已切换到 `echarts/core + ReactEChartsCore`。
- `src/components/ECharts/DeferredEChart.tsx` 会先渲染图表占位，再异步加载真实图表组件。
- 当前只按需注册以下图表与组件：
  - 图表：`LineChart / BarChart / GaugeChart / PieChart / RadarChart`
  - 组件：`Title / Tooltip / Grid / Legend / Radar / Dataset / Transform / CanvasRenderer`
- 这样避免了全量 ECharts 能力进入同一个 vendor 包。
- `vite.config.ts` 已新增构建分包策略：
  - 路由页面使用 `lazy + Suspense`
  - `mqtt`、`echarts`、导出依赖等重模块分离成独立 vendor chunk
- `src/utils/mqttClient.ts` 已改为运行时动态 `import('mqtt')`：
  - 未进入 Dashboard/Monitor 的 MQTT 连接流程前，不会预先拉取 MQTT 浏览器库
  - 这样优化的是加载时机，而不是单纯追求把 `vendor-mqtt` 文件名从产物里消掉
- Dashboard / History / Monitor / Screen 的图表已进一步切到 `DeferredEChart`：
  - 页面可见结构先出来
  - 图表组件和 ECharts 初始化随后异步进行
  - 优化的是首屏可交互和可见内容到达时间，不一定直接体现为 build 产物数字继续明显下降
- `SiteModal` 在 `Dashboard` 与 `Sites` 中已改为 `React.lazy` 按需加载：
  - 站点列表页和看板页初始进入时，不再把站点配置大弹窗作为静态同步依赖
  - 只有真正点击“配置此站点 / 编辑配置 / 新建站点”时才加载该 chunk
- `History` 页面已继续拆分：
  - `HistoryTableCard` 独立承载大表格
  - `HistoryReportModal` 独立承载报告生成弹窗
  - 历史页现在是“筛选与图表主体先渲染，表格和报告逻辑后加载”的结构
- 目前构建结果相较上一阶段已经明显改善：
  - 原先单一主包约 3MB 级别
  - 现在页面逻辑已拆成多个路由 chunk
  - ECharts vendor 包已从约 `1.13MB` 压到约 `598KB`
- 仍然存在的剩余大包：
  - `vendor-echarts`
  - `vendor-mqtt`
  - `table/date-picker/vendor-react` 等通用依赖
- 下一阶段如果要继续压缩，需要进一步减少图表页首次依赖、评估 MQTT 浏览器端 polyfill 开销，并审视 Ant Design 重组件的加载边界。

### 5.6 Dashboard 与 Monitor 联动逻辑

- `Dashboard`：按 `deviceId` 匹配当前站点传感器，将 MQTT 实时值映射到 Gauge。
  - 站点切换、站点保存回刷、植物生理实时覆盖已封装在 `src/pages/Dashboard/useDashboardRuntime.ts`。
  - 遥测解析与图表配置从页面本体移到 `src/pages/Dashboard/dashboardShared.tsx`，减少 `Dashboard.tsx` 内联逻辑。
- `Monitor`：
  - 接收传感器数据并刷新卡片与历史序列（最近 20 点）。
  - 下发控制指令后等待 ack（超时 10s），维护指令状态机（发送中/已确认/超时/失败）。
  - 高频页面状态已迁到 `src/stores/monitorStore.ts`，页面自身只保留 MQTT 订阅、超时清理、模拟器定时器等副作用控制。
  - 视图层已拆到 `src/pages/Monitor/components/`，共享格式化与排序逻辑收口在 `src/pages/Monitor/monitorViewShared.ts`。
  - 运行时副作用编排进一步封装进 `src/pages/Monitor/useMonitorRuntime.ts`，并用 `Map` 缓存 `deviceId -> sensor` 映射，降低高频消息处理时的重复查找开销。
- `Dashboard` 与 `Monitor` 现已共用 `src/utils/mqttTelemetry.ts`：
  - 统一解析 MQTT payload
  - 统一提取 `deviceId`
  - 统一读取数值型遥测值
  - 统一构建设备映射表
- `Dashboard / Monitor / Map / Sites / IoT` 现已共用 `src/stores/siteStore.ts`：
  - 统一读取站点列表
  - 统一维护当前站点
  - 统一响应站点保存/删除/切换
  - `siteRepository.ts` 负责数据访问抽象，`siteStorage.ts` 负责当前持久化实现，`siteStore.ts` 负责运行时联动
- 支持本地模拟器实现自收自发闭环演示（演示时无需真实硬件）。

### 5.7 Monitor 状态治理策略

- 本轮引入 Zustand，但没有把所有运行时对象一股脑搬进 store。
- **已进 store**：`mqttStatus`、`broker`、`messageCount`、`logs`、`sensorStateMap`、`deviceCommandState`、`commandHistory`、`simulator`、筛选状态。
- **仍留在组件层**：`ackWaitersRef`、模拟器 interval、阀门定时开启 minutes ref、`nowTick` 计时器，以及 MQTT 订阅/初始化 effect。
- 原因是这些对象本质上属于生命周期和资源管理，而不是业务可序列化状态；强行放进 store 会让清理、超时和闭包一致性更差。
- 当前结构已经明确分层：
  - `monitorStore.ts`：可序列化实时态
  - `useMonitorRuntime.ts`：运行时副作用与命令链路编排
  - `Monitor.tsx`：站点上下文解析与面板装配
  - `components/`：纯视图面板
  - `monitorViewShared.ts`：视图共享格式化与常量
  - `monitorRuntimeShared.ts`：运行时共享方法与 Topic 拼装

### 5.8 站点编辑器当前性能策略

- `SiteModal` 已完成第一阶段拆分：父组件只负责初始化、状态编排、保存和跨步骤联动，渲染细节下放到 `SiteBasicInfoStep`、`SiteFieldEditorStep`、`SiteDecisionModeStep`、`SiteAlarmRulesStep`。
- 田块编辑器拖拽更新已改成 `requestAnimationFrame` 按帧批处理；鼠标移动时先写入 Ref，再在动画帧内统一提交 React State。
- 共享常量、报警定义、传感器枚举已收口到 `src/pages/Sites/components/siteModalShared.ts`，减少多处重复维护。
- 下一阶段如果画布复杂度继续上升，应把第 2 步中的交互状态继续下沉，必要时再评估 Canvas/Konva 渲染方案。

### 5.9 决策引擎逻辑（前端模拟）

`/engine` 页面的 5 种决策模式均用前端 JS 实现：

- **模式1 定时灌溉**：按时间窗口判断。
- **模式2 ET₀计算（FAO-56）**：`ETc - rainDiscount × 降雨量 > 0` 时触发。
- **模式3 土壤含水率阈值**：低于下限开灌，高于上限停灌。
- **模式4 土壤水势阈值**：水势低于起始阈值开灌。
- **模式5 植物水分亏缺指标**：液流/茎径/膨压加权评分 < 60 分触发（核心卖点）。

后续应将此逻辑迁移至后端，作为可调参的决策引擎服务。

## 7. 已有能力与商业化差距（客观评估）

### 7.1 已具备

- 多页面业务框架完整（11 个页面）。
- 站点配置闭环可跑通（repository + localStorage 持久化）。

6. **知识库数据单薄**：植物参数表目前为 mock，须引入席老师团队真实实验数据。
7. **高德地图未接入**：`/map` 页面当前为 SVG 示意版本，需配置真实 API Key。

## 8. 下一步开发建议（分优先级）

### P0（先做，商业化必需）

1. **后端落地**（FastAPI 推荐，与 Python 决策引擎生态一致）
   - 表：站点、设备、传感器数据（时序）、告警、控制指令、用户权限。
   - 数据库：PostgreSQL + TimescaleDB（时序数据）。
   - 前端优先完成 `siteRepository` 的本地实现 → API 实现替换。
2. **MQTT 生产化**
   - 私有 Broker 部署（EMQ X 或 Mosquitto）。
   - 设备鉴权（token/证书）+ Topic ACL。
   - QoS 策略、重连策略、离线消息缓存。
3. **知识库数据录入**
   - 将席老师团队积累的植物水分参数（毛白杨、苹果等）录入正式知识库。
   - 这是产品区别于竞品的核心数据资产。

### P1（稳定性与交付质量）

1. **测试体系**：单测（mqttClient、决策函数）+ 组件测试 + E2E。
2. **状态管理**：把 Zustand 继续扩展到 Dashboard 等页面，统一实时态与跨页设备态。
3. **继续压缩前端包体积**：
   - 审视 `vendor-mqtt` 的浏览器端开销
   - 评估历史数据 / 大屏等图表重页面的进一步延迟加载
   - 必要时减少 Ant Design 重组件的公共首依赖
4. **样式治理**：建立统一 Design Token 映射，清理硬编码色值。
5. **高德地图接入**：`/map` 升级为真实卫星地图，显示 22 个 CP-GPE 站点。

### P2（产品化增强）

1. 多租户与角色权限（管理员/运维/只读）。
2. 策略编排与规则引擎可视化（灌溉计划自动生成）。
3. 数据大屏支持站点切换与模板管理。
4. 国际化 i18n 架构（目前为静态中英并排，非标准 i18n）。
5. 移动端 App（React Native / Expo）。

## 11. 结论

当前项目已从"页面展示型 demo"提升为"具备站点配置与实时通信闭环的前端原型平台"。

下一阶段重点不是继续堆页面，而是：

- **把数据与控制链路服务化**（后端 + 鉴权 + 存储）；
- **把核心知识库数据化**（席老师团队实测数据入库，构建真正的技术壁垒）；
- **把工程质量体系补齐**（测试 + 监控 + 文档一致性）；
- **把视觉与交互标准固化**（统一 token 与组件规范）。

只有完成以上四点，才能稳定进入可售卖的商业化产品阶段。
